0. 시작하기 — 변수 · 타입 · 문자열

📖 이 장의 학습 목표 이 장을 마치면 다음을 할 수 있습니다.

  • ☑️ 타입(형) 과 데이터 객체(변수) 의 차이를 설명하고, TYPES·DATA를 구분해 선언할 수 있다.
  • ☑️ TYPE(글로벌 타입 참조)과 LIKE(기존 변수 참조)를 상황에 맞게 쓸 수 있다.
  • ☑️ CONCATENATE·SPLIT·오프셋 표기(+n(m))로 문자열을 조작할 수 있다.
  • ☑️ cl_abap_char_utilities의 개행 상수로 여러 줄 텍스트를 저장·복원하는 흐름을 이해한다.

들어가기 전에. ABAP는 SAP ERP를 움직이는 언어입니다. 다른 언어를 다뤄 봤다면 문법 자체는 낯설지 않지만, ABAP만의 두 가지 관례를 먼저 익히면 이후가 쉬워집니다. 하나는 "틀(타입)과 실물(변수)을 엄격히 구분" 한다는 점이고, 다른 하나는 대부분의 타입을 데이터베이스 필드에서 빌려 온다(TYPE sflight-connid 같은)는 점입니다. 이 장에서 그 감각부터 잡습니다.

0.1 변수와 타입 — TYPES vs DATA

데이터를 다루려면 먼저 "틀(형)" 과 "실제 값(데이터 객체)" 을 구분해야 합니다. 이 구분은 ABAP 전반에서 반복해서 등장하므로 처음부터 확실히 잡아 두는 것이 좋습니다.

  • TYPES : 데이터의 틀(형)만 정의합니다. 메모리를 잡지 않으며 그 자체에는 값을 넣을 수 없습니다. → 붕어빵 틀
  • DATA : 실제 변수(데이터 객체) 를 선언합니다. 메모리를 확보하고 값을 대입할 수 있습니다. → 붕어빵
DATA  name_d TYPE c.    " name_d = 'A'  가능 (실제 변수)
TYPES name_t TYPE c.    " name_t = 'A'  불가 (타입일 뿐)

참조 선언에서 TYPE은 글로벌 타입(Data Element·Dictionary 구조)을, LIKE는 이미 선언된 다른 변수(로컬)를 참조합니다. 로컬 구조체 타입을 TYPES로 만들고, 그 타입으로 구조체와 내부 테이블을 선언하는 것이 ABAP의 가장 기본적인 패턴입니다(1장에서 본격적으로 다룹니다).

TYPES: BEGIN OF gty_s_flightocc,
         connid   TYPE sflight-connid,
         fldate   TYPE sflight-fldate,
         seatsocc TYPE sflight-seatsocc,
       END OF gty_s_flightocc.

DATA gt_flightocc TYPE TABLE OF gty_s_flightocc.  " 내부 테이블
DATA gs_flightocc TYPE gty_s_flightocc.           " 구조체

📌 개념 — 왜 DB 필드에서 타입을 빌릴까? TYPE sflight-connid처럼 딕셔너리 필드를 참조하면, 그 필드의 길이·소수 자릿수·의미가 그대로 따라옵니다. 나중에 원본 필드가 바뀌어도 코드가 자동으로 맞춰지므로, 실무에서는 숫자 리터럴로 길이를 직접 적기보다 딕셔너리 참조를 선호합니다.

0.2 문자열 처리

업무 프로그램은 코드값을 이어 붙이거나, 한 줄의 텍스트를 조각내는 일이 잦습니다. ABAP가 제공하는 문자열 도구를 익혀 둡니다.

반복문으로 문자열 쌓기

DO ... ENDDO 반복문과 CONCATENATE로 문자열을 한 글자씩 늘려 갈 수 있습니다. 아래는 입력한 수만큼 별(*)을 늘려 그리는 예입니다.

PARAMETERS: pa_num1 TYPE i.
DATA: gv_result TYPE string.

DO pa_num1 TIMES.
  CONCATENATE '*' gv_result INTO gv_result.
  WRITE: gv_result, /.
ENDDO.

오프셋·길이로 잘라내기

문자열+n(m) 표기는 n번째 다음 문자부터(0부터 셈) m글자를 잘라냅니다. 날짜 타입 d도 내부적으로는 8자리 문자열(YYYYMMDD)이라 같은 방식으로 연·월·일을 뽑아낼 수 있습니다.

DATA: gv_char TYPE string VALUE '12345678'.
WRITE: gv_char+2(2).   " '34' — 앞 2글자를 건너뛰고 2글자

오프셋·길이 예제 실행 결과 — gv_char+2(2)가 '34'를 출력함.

합치기와 나누기 — CONCATENATE / SPLIT

여러 문자열을 하나로 합칠 때는 CONCATENATE(구분자는 SEPARATED BY), 하나를 여러 조각으로 나눌 때는 SPLIT ... AT ... INTO를 씁니다.

DATA : lv_text(20), lv_char1(2), lv_char2(2), lv_char3(2).

CONCATENATE 'AABDS' 'ddkedwkd' 'iekfkdsf'
  INTO lv_text SEPARATED BY ','.

SPLIT 'A,B,C' AT ',' INTO lv_char1 lv_char2 lv_char3.

여러 줄 텍스트 다루기 — 개행 상수 cl_abap_char_utilities

풀어야 할 문제. 화면의 텍스트 에디터는 여러 줄을 입력받지만, 데이터베이스 필드는 하나의 문자열 한 칸입니다. 그래서 "여러 줄 ↔ 한 필드"를 서로 바꿔 주어야 하는데, 이때 줄과 줄 사이를 구분하는 줄바꿈 문자가 필요합니다. 이 줄바꿈을 눈에 보이는 상수로 다루는 것이 cl_abap_char_utilities=>newline입니다(cr_lf는 CR+LF). 아래는 실제 텍스트 에디터 화면입니다 — 사용자가 다섯 줄을 입력하고 SAVE를 누르는 모습입니다.

텍스트 에디터에 여러 줄(1st~5st)을 입력하고 SAVE/CLEAR로 저장·초기화하는 화면.

개행 상수의 정체. newline은 눈에 보이지 않는 한 글자(줄바꿈)이며, 내부 값은 0A00, 디버거에서는 #로 표시됩니다. 이 # 하나가 "여기서 줄을 바꾼다"는 표시입니다.

CL_ABAP_CHAR_UTILITIES=>NEWLINE 상수의 실제 값(# = 0A00).

① 저장 — 여러 줄을 한 필드로 합치기 (CONCATENATE)

에디터에서 읽어 온 각 줄 사이에 newline을 끼워 넣어 하나의 필드로 이어 붙인 뒤 DB에 저장합니다.

* (저장) 인터널 테이블 → 한 필드로 합치기(줄바꿈 유지): CONCATENATE
FORM save_description.
  DATA: ls_save TYPE zc2010001,
        BEGIN OF ls_line, line TYPE tdline, END OF ls_line,
        lt_line LIKE TABLE OF ls_line.

  CALL METHOD gcl_editor->get_text_as_r3table IMPORTING table = lt_line.

  ls_save-carrid = gs_schedule-carrid.
  ls_save-connid = gs_schedule-connid.
  LOOP AT lt_line INTO ls_line.
    CONCATENATE ls_save-description ls_line-line cl_abap_char_utilities=>newline
      INTO ls_save-description.
  ENDLOOP.

  MODIFY zc2010001 FROM ls_save.
  IF sy-dbcnt > 0.
    COMMIT WORK AND WAIT.
    MESSAGE s000 WITH text-m03.
  ENDIF.
ENDFORM.

단계별로 디버거에서 확인하면 이렇습니다. 먼저 get_text_as_r3table로 에디터 내용이 라인별 내부 테이블 LT_LINE(5줄)에 담깁니다.

에디터 → 내부 테이블 LT_LINE: 각 줄(1st~5st)이 한 행씩 담깁니다.

이어서 LOOP를 돌며 각 줄 뒤에 newline을 붙여 CONCATENATE합니다.

CONCATENATE ... cl_abap_char_utilities=>newline — 줄바꿈을 끼워 한 필드로 합치는 코드.

그 결과, 다섯 줄이 1st#2st#3st#4st#5st#처럼 #(newline)로 이어진 한 필드 LS_SAVE-DESCRIPTION가 됩니다. 이 값이 DB에 저장됩니다.

합쳐진 결과: DESCRIPTION = 1st#2st#3st#4st#5st# (한 필드에 여러 줄이 newline으로 저장).

② 복원 — 한 필드를 여러 줄로 되돌리기 (SPLIT)

화면을 다시 열 때는 반대로, 저장된 한 필드를 newline 기준으로 잘라 라인별 내부 테이블로 만든 뒤 에디터에 넣습니다.

* (복원) 필드 텍스트 → 인터널 테이블(라인별): SPLIT
SPLIT ls_desc-description AT cl_abap_char_utilities=>newline INTO TABLE lt_line.
CALL METHOD gcl_editor->set_text_as_r3table EXPORTING table = lt_line.

DB에서 읽어 온 필드는 저장 때와 같이 #(newline)가 섞인 한 덩어리입니다.

저장된 한 필드 LS_DESC-DESCRIPTION — 여러 줄이 #로 이어져 있습니다.

이 값을 SPLIT ... AT newline으로 잘라 냅니다.

SPLIT ... AT cl_abap_char_utilities=>newline INTO TABLE lt_line — 한 필드를 라인별로 분리하는 코드.

그 결과 LT_LINE이 다시 라인별 여러 행으로 나뉘고, set_text_as_r3table로 에디터에 채워 원래 모습대로 복원됩니다.

분리 결과: 한 필드가 다시 라인별 내부 테이블 LT_LINE로 나뉩니다.

📌 정리 — 핵심은 하나입니다. 저장은 CONCATENATE로 newline을 끼워 여러 줄 → 한 필드, 복원은 SPLIT으로 newline을 기준 삼아 한 필드 → 여러 줄. 이 왕복 패턴은 8장 화면 프로그램의 텍스트 에디터에서 다시 만납니다.

0.3 이 장의 요약

ABAP에서는 타입(TYPES, 틀) 과 변수(DATA, 실물) 를 엄격히 구분하며, 타입은 되도록 딕셔너리 필드(TYPE 테이블-필드)에서 빌려 오는 것이 관례입니다. 참조는 글로벌 타입엔 TYPE, 기존 변수엔 LIKE를 씁니다. 문자열은 CONCATENATE로 합치고 SPLIT으로 나누며, +n(m) 오프셋 표기로 일부를 잘라냅니다. 줄바꿈이 든 긴 텍스트는 cl_abap_char_utilities의 개행 상수를 매개로 저장·복원합니다.

✅ 확인 문제

  1. TYPES name_t TYPE c. 로 선언한 name_t에 값을 대입할 수 없는 이유는 무엇인가요?
  2. TYPE과 LIKE는 각각 무엇을 참조하나요?
  3. gv_char가 '12345678'일 때 gv_char+2(2)의 결과는 무엇인가요?
  4. 여러 줄 텍스트를 한 DB 필드에 저장했다가 라인별로 되살리려면 어떤 명령과 상수를 쓰나요?
정답 보기
  1. TYPES는 "틀(형)"만 정의할 뿐 메모리를 확보하지 않는 타입이기 때문입니다. 값을 담으려면 DATA로 실제 변수를 선언해야 합니다.
  2. TYPE은 글로벌 타입(Data Element·Dictionary 구조)을, LIKE는 이미 선언된 다른 변수(로컬)를 참조합니다.
  3. '34' — 앞 2글자를 건너뛰고 그다음 2글자를 잘라냅니다.
  4. 저장은 CONCATENATE로 각 줄 사이에 cl_abap_char_utilities=>newline을 끼워 한 필드로 합치고, 복원은 SPLIT ... AT cl_abap_char_utilities=>newline INTO TABLE로 라인별로 나눕니다.

1. 데이터 다루기 — 구조체와 내부 테이블

📖 이 장의 학습 목표 이 장을 마치면 다음을 할 수 있습니다.

  • ☑️ 여러 필드를 하나의 레코드로 묶는 구조체(Structure) 를 선언하고, INCLUDE로 재사용할 수 있다.
  • ☑️ 같은 구조의 행을 여러 개 담는 내부 테이블(Internal Table) 을 선언하고, 행을 추가·조회·수정할 수 있다.
  • ☑️ SORT → DELETE ADJACENT DUPLICATES → BINARY SEARCH가 왜 한 묶음으로 움직이는지 이해하고 올바른 순서로 쓸 수 있다.
  • ☑️ 워크에어리어(WA) 와 내부 테이블의 관계, CLEAR와 REFRESH의 차이를 설명할 수 있다.

들어가기 전에. 0장에서 변수 하나에 값 하나를 담는 법을 익혔습니다. 하지만 업무 데이터는 "한 줄에 여러 칸"(예: 전표 한 건 = 회사코드 + 전표번호 + 연도 + 금액)이고, 그런 줄이 "여러 개"(전표 목록) 모여 있습니다. 이 두 가지 모양을 ABAP에서 표현하는 도구가 바로 구조체(한 줄)와 내부 테이블(여러 줄)입니다. 다른 언어의 struct/record와 array/list에 해당한다고 보면 되지만, ABAP에서는 이 둘이 데이터베이스 테이블과 자연스럽게 대응된다는 점이 핵심입니다.

1.1 구조체 — 여러 필드를 하나로 묶기

구조체는 서로 다른 타입의 필드 여러 개를 하나의 이름으로 묶은 레코드입니다. 전표 한 건, 자재 한 개, 사원 한 명처럼 "현실의 한 대상"을 코드에서 한 덩어리로 다루고 싶을 때 씁니다. C의 struct, ABAP 이전 세대의 개념과 같지만, ABAP에서는 데이터베이스 테이블의 한 행(row)과 모양이 그대로 맞아떨어지도록 설계하는 것이 관례입니다.

선언은 TYPES: BEGIN OF ... END OF(타입만 정의)와 DATA: BEGIN OF ... END OF(실제 변수 생성) 두 방식이 있습니다. 재사용할 모양은 TYPES로 정의해 두고, 실제로 값을 담을 그릇은 DATA로 만드는 것이 좋은 습관입니다.

📌 개념 — TYPES vs DATA TYPES는 "설계도(모양)"만 만들고 메모리를 차지하지 않습니다. DATA는 그 설계도로 "실물(변수)"을 찍어내 메모리를 확보합니다. 0장의 이 구분이 구조체에서도 그대로 적용됩니다.

INCLUDE로 구조체 재사용하기

이미 만들어진 구조체나 딕셔너리 구조를 중간에 끼워 넣어 새 구조체를 조립할 수 있습니다. 같은 필드 묶음을 여러 곳에서 반복 선언하지 않게 해 주는, 재사용의 핵심 도구입니다. INCLUDE STRUCTURE(딕셔너리 구조)와 INCLUDE TYPE(로컬 타입)을 상황에 맞게 씁니다.

TYPES: BEGIN OF lty_list,
         gjahr TYPE bkpf-gjahr,
         mtart TYPE mara-mtart,
       END OF lty_list.

DATA: BEGIN OF ls_data.
        INCLUDE STRUCTURE fac_bkpf_ext_glo.
DATA:   bukrs TYPE t001-bukrs,
        werks TYPE t001w-werks.
        INCLUDE STRUCTURE emarc.
        INCLUDE TYPE lty_list.
DATA:   maktx TYPE makt-maktx,
      END OF ls_data,
      lt_data LIKE TABLE OF ls_data.

위 선언 한 번으로 fac_bkpf_ext_glo·emarc의 모든 필드와 직접 적은 bukrs·werks·maktx, 그리고 lty_list의 필드까지 한 구조체 안에 전부 들어옵니다. 마지막 줄 lt_data LIKE TABLE OF ls_data는 이 구조체를 행으로 갖는 내부 테이블까지 한 번에 만들어 둔 것입니다(내부 테이블은 1.2에서 다룹니다).

인클루드된 구조체의 필드는 따로 선언하지 않아도 자동으로 포함되고, 접근할 때 상위 이름을 생략할 수 있습니다. 예를 들어 address 구조를 인클루드했다면 wa-address-street가 아니라 wa-street로 바로 접근됩니다.

DATA wa_person TYPE zperson_cl2_01.
wa_person-name-firstname = 'harry'.        " name 은 인클루드된 구조체
wa_person-street         = 'Privet Drive'. " address 를 생략하고 접근

SE11 데이터 엘리먼트(ZFIRSTNAME_CL2_01) 화면 — INCLUDE로 재사용할 딕셔너리 필드의 도메인/타입 정의

SE11 구조체(ZPERSON_CL2_01) Components 탭 — .INCLUDE로 조립해 상속 필드가 자동 포함됨

.INCLUDE로 만든 구조체는 직접 선언하지 않은 필드까지 자동으로 포함합니다.

구조체 안에는 단순 필드뿐 아니라 테이블 타입도 컴포넌트로 넣을 수 있습니다. "사원 한 명(구조체) 안에 전화번호 여러 개(내부 테이블)"처럼 1:N 관계를 한 덩어리로 표현할 때 유용합니다.

DATA wa_phone TYPE str_phone.
wa_phone-p_type   = 'p'.
wa_phone-p_number = '+31-10-993890'.
INSERT wa_phone INTO TABLE wa_person-phone.   " 구조체 내부 테이블에 추가
LOOP AT wa_person-phone INTO wa_phone.
  WRITE: AT 20 wa_phone-p_type, wa_phone-p_number, /.
ENDLOOP.

gs_flight 워크에어리어(carrid·connid·fldate·seatsmax·seatsocc·percentage)를 한 줄 레코드로 나타낸 개념 도식

MOVE-CORRESPONDING 개념 도식 — 이름이 같은 필드끼리만 값이 복사됨을 화살표로 표현

1.2 내부 테이블 — 같은 구조의 행을 여러 개 담기

내부 테이블은 같은 구조의 행을 여러 개 담는 메모리 상의 테이블입니다. 데이터베이스에서 읽어온 여러 건을 담아 두고 반복 처리하거나, 화면에 목록으로 뿌리기 직전까지 데이터를 가공하는 데 거의 모든 프로그램에서 쓰입니다. ABAP에서 가장 자주 다루게 될 자료구조입니다.

핵심은 내부 테이블은 "여러 줄", 워크에어리어(WA)는 "지금 다루는 한 줄" 이라는 관계입니다. 내부 테이블에 값을 넣거나 꺼낼 때는 항상 그 행과 같은 구조의 워크에어리어를 거칩니다. 테이블을 서랍장, 워크에어리어를 작업대 위에 꺼내 놓은 서랍 한 칸이라고 생각하면 이해가 쉽습니다.

📌 명명 관례 — 실무에서 워크에어리어는 ls_(local structure) 또는 ws_/gs_, 내부 테이블은 lt_(local table) 또는 gt_ 접두어를 붙입니다. 이 노트도 그 관례를 따릅니다.

행 추가 — APPEND vs INSERT

  • APPEND wa TO itab : 맨 뒤에 한 줄 덧붙입니다. 순서가 중요한 일반(Standard) 테이블에서 가장 흔합니다.
  • INSERT wa INTO TABLE itab : 테이블 종류에 맞는 위치에 넣습니다(정렬 테이블이면 키 순서 자리에 삽입). 유일 키 위반이면 sy-subrc로 알려 줍니다.

행 조회 — READ TABLE

  • 조건 조회 : READ TABLE itab INTO wa WITH KEY 필드 = 값 ...
  • 위치 조회 : READ TABLE itab INTO wa INDEX n
  • 어느 쪽이든 성공/실패는 sy-subrc(0이면 성공) 로 판단합니다. 찾지 못했는데 워크에어리어를 그대로 쓰면 이전 값이 남아 있으니 반드시 sy-subrc를 확인하세요.

행 수정 — MODIFY

  • 위치 기준 수정은 MODIFY itab FROM wa INDEX sy-tabix, 키 기준 수정은 MODIFY TABLE itab FROM wa 를 씁니다.
  • 특정 필드만 바꾸려면 TRANSPORTING 필드1 필드2 를 덧붙입니다.

아래는 추가와 조회를 함께 보여 주는 예입니다.

DATA : ls_header TYPE zc2s010001,
       lt_header LIKE TABLE OF ls_header.

ls_header-bukrs = '100'. ls_header-belnr = 'aaaa'.
APPEND ls_header TO lt_header.  CLEAR ls_header.

ls_header-bukrs = '200'. ls_header-belnr = 'bbbb'.
INSERT ls_header INTO TABLE lt_header.  CLEAR ls_header.

READ TABLE lt_header INTO ls_header WITH KEY bukrs = '100' belnr = 'aaaa'.  " 조건 조회
READ TABLE lt_header INTO ls_header INDEX 3.                                " 위치 조회

LOOP AT lt_header INTO ls_header.
  WRITE: / ls_header-bukrs, ls_header-belnr.
ENDLOOP.

💡 팁 — APPEND 뒤의 CLEAR : 워크에어리어에 값을 담아 APPEND한 뒤에는 CLEAR ls_header로 비워 주는 습관을 들이세요. 그러지 않으면 다음 행에 이전 값이 섞여 들어가는 실수가 잦습니다.

다음은 데이터베이스에서 읽어온 뒤 특정 필드만 골라 수정하는 실전 패턴입니다. SELECT으로 내부 테이블에 담고(2장에서 자세히 다룹니다), 루프를 돌며 값을 바꾼 뒤 MODIFY TABLE ... TRANSPORTING 으로 반영합니다.

DATA : BEGIN OF ls_data,
         carrid    TYPE sflight-carrid,
         connid    TYPE sflight-connid,
         fldate    TYPE sflight-fldate,
         currency  TYPE sflight-currency,
         planetype TYPE sflight-planetype,
         seats     TYPE sflight-seatsocc_b,
       END OF ls_data,
       lt_data LIKE TABLE OF ls_data.

SELECT carrid connid fldate currency planetype seatsocc_b
  FROM sflight INTO TABLE @lt_data
  WHERE currency = 'USD' AND planetype = '747-400'.

LOOP AT lt_data INTO ls_data.
  ls_data-seats = ls_data-seats + 5.
  MODIFY TABLE lt_data FROM ls_data TRANSPORTING seats.   " 키 기준 특정 필드 수정
ENDLOOP.
cl_demo_output=>display( lt_data ).

💡 팁 — SELECT ... ENDSELECT로 한 건씩 받아 처리할 수도 있지만, 내부 테이블에 한 번에 담아 LOOP를 도는 방식이 성능·가독성 면에서 표준입니다. DB 왕복을 줄이는 것이 실무의 기본기입니다.

1.3 정렬 · 중복 제거 · 이진 검색

이 세 명령은 겉보기엔 따로지만 실제로는 한 묶음으로 움직입니다. 공통 전제는 SORT가 먼저 실행되어 테이블이 정렬돼 있어야 한다는 것이고, 각 명령에서 비교하는 필드 순서가 정렬 순서와 일치해야 결과가 정확합니다.

  • SORT : 지정한 필드 기준으로 정렬합니다(필드마다 ASCENDING/DESCENDING 개별 지정 가능).
  • DELETE ADJACENT DUPLICATES ... COMPARING : 인접한 중복 행만 지웁니다. 그래서 반드시 먼저 정렬해 같은 값을 붙여 놓아야 합니다("ADJACENT"=인접, 이 단어가 곧 SORT가 선행돼야 하는 이유입니다).
  • READ TABLE ... BINARY SEARCH : 정렬된 상태를 전제로 이진 검색을 해 대량 데이터에서 조회를 훨씬 빠르게 합니다.
SORT lt_data BY carrid connid fldate.     " 필드별 ASC/DESC 개별 지정 가능
DELETE ADJACENT DUPLICATES FROM lt_data COMPARING carrid connid fldate.

READ TABLE lt_data INTO ls_data
     WITH KEY carrid = pa_carr connid = 17 BINARY SEARCH.   " 정렬 상태에서 이진 검색

⚠️ 주의 — BINARY SEARCH를 정렬되지 않은 테이블에 쓰면 있는 값도 못 찾을 수 있습니다. 정렬 필드와 검색 키의 순서가 어긋나도 마찬가지입니다. "정렬 → 중복 제거 → 이진 검색"을 세트로 기억하세요.

1.4 초기화와 자주 쓰는 도구들

마지막으로 내부 테이블을 다룰 때 늘 함께 등장하는 보조 도구들입니다.

  • 초기화 — CLEAR vs REFRESH : CLEAR는 변수·구조체·테이블 무엇이든 내용을 비웁니다. REFRESH는 내부 테이블 전용으로 모든 행을 지웁니다. 헤더 라인이 있는 테이블은 바디 []까지 명시해야 완전히 비워집니다.
  • 헤더 라인 테이블(WITH HEADER LINE, OCCURS 0) : 워크에어리어가 테이블에 내장돼 있던 옛 방식입니다. 신규 개발에서는 별도 워크에어리어를 쓰는 것이 권장되지만, 기존 코드를 읽으려면 알아 둬야 합니다.
  • TABLES: <t>. : 딕셔너리 테이블과 같은 구조의 전역 워크에어리어를 만듭니다(주로 화면 입력 필드용). 역시 오래된 문법이지만 화면 프로그램(8장)에서 여전히 만납니다.
  • MOVE-CORRESPONDING : 두 구조체에서 이름이 같은 필드끼리만 값을 복사합니다. 구조가 다른 두 데이터를 합칠 때 아주 유용합니다.
CLEAR lt_data.     " 내용 초기화
REFRESH lt_data.   " 내부 테이블 초기화

DATA lt_mara TYPE TABLE OF mara WITH HEADER LINE.
lt_mara-matnr = 'AAA'.
APPEND lt_mara.               " 헤더를 바디에 추가
CLEAR: lt_mara, lt_mara[].    " 헤더 + 바디 모두 초기화

TABLES: sflight.
sflight-carrid = 'AA'.

MOVE-CORRESPONDING gs_carrier TO gs_carrierflight.

내부 테이블에서 READ/LOOP한 값('AA')을 출력한 WRITE 리스트 실행 결과

1.5 이 장의 요약

구조체는 "한 줄", 내부 테이블은 "여러 줄", 워크에어리어는 "지금 다루는 한 줄"입니다. 내부 테이블에 값을 넣고 꺼내는 일은 언제나 워크에어리어를 통해 이뤄집니다. 행 추가는 APPEND/INSERT, 조회는 READ TABLE(성공 여부는 sy-subrc로 확인), 수정은 MODIFY(특정 필드는 TRANSPORTING)로 합니다. 정렬·중복제거·이진검색은 SORT를 전제로 한 세트로 움직이며, 초기화는 상황에 따라 CLEAR와 REFRESH를 구분해 씁니다. 이 도구들은 이후 모든 장에서 계속 쓰이니 손에 익혀 두는 것이 좋습니다.

✅ 확인 문제

  1. APPEND와 INSERT ... INTO TABLE의 차이는 무엇이며, 정렬된(Sorted) 테이블에는 어느 쪽을 쓰는 것이 맞을까요?
  2. READ TABLE로 조회했는데 원하는 값이 없을 때, 그것을 어떻게 알 수 있나요?
  3. DELETE ADJACENT DUPLICATES 앞에 반드시 SORT가 필요한 이유를 "ADJACENT"라는 단어로 설명해 보세요.
  4. CLEAR와 REFRESH는 각각 무엇을 비우며, 헤더 라인 테이블을 완전히 비우려면 어떻게 써야 하나요?
정답 보기
  1. APPEND는 항상 맨 뒤에 붙이고, INSERT ... INTO TABLE은 테이블 종류에 맞는 위치(정렬 테이블이면 키 순서 자리)에 넣습니다. 정렬 상태를 유지해야 하는 Sorted 테이블에는 INSERT ... INTO TABLE이 맞습니다.
  2. sy-subrc가 0이 아니면 조회 실패입니다. 워크에어리어에 이전 값이 남아 있을 수 있으므로 반드시 sy-subrc로 판단합니다.
  3. DELETE ADJACENT DUPLICATES는 인접한(ADJACENT) 행만 비교해 지웁니다. 같은 값이 서로 떨어져 있으면 지워지지 않으므로, 먼저 SORT로 같은 값을 붙여 놓아야 합니다.
  4. CLEAR는 대상의 내용을(변수·구조체·테이블 모두) 비우고, REFRESH는 내부 테이블의 모든 행을 지웁니다. 헤더 라인 테이블은 CLEAR: itab, itab[]. 처럼 헤더와 바디 []를 함께 비워야 완전히 초기화됩니다.
